# 每日一條龍 (Daily Loop) 流程深度對比報告

本報告詳細羅列了當前 **Estrella-071 優化版** 與 **上游 (Upstream) 正式版** 在執行每日任務（一條龍）時的邏輯差異。

## 核心差異總覽

| 維度 | 上游正式版 (Standard) | Estrella-071 優化版 (High-Speed) |
| :--- | :--- | :--- |
| **執行架構** | 串行阻塞式 (Serial Blocking) | 並行非阻塞式 (Parallel Async) |
| **數據讀取** | 實時 OCR，讀取一個值點擊一次，全程等待 | 極速連拍快照，進入加載畫面後背景分析 |
| **UI 交互** | 依賴固定休眠 (`after_sleep`) | 依賴指令響應間隙，毫秒級切換 |
| **加載利用率** | 100% 浪費（原地等待黑屏結束） | 100% 利用（在黑屏時完成所有 OCR 分析） |

---

## 詳細步驟對比

### 第一階段：啟動與資源定位

| 步驟 | 上游正式版流程 (Standard) | Estrella-071 極速版 (Optimized) |
| :--- | :--- | :--- |
| **1. 首要行動** | **立即傳送到深塔** (不管數據)。 | **啟動每日數據採集** (連拍快照)。 |
| **2. 第一次加載** | 阻塞等待地圖加載 (約 15-30s)。 | (此時仍在書本介面，極速快照中)。 |
| **3. 數據讀取** | **在深塔下圖後**再次開書執行 OCR。 | **直接從當前書本**切換分頁發起傳送。 |
| **4. 耗時感官** | 等待完第一次傳送才開始讀數據。 | 先讀完數據，在**唯一的傳送加載中**分析。 |

### 第二階段：地圖跳轉與並行化處理

| 步驟 | 上游正式版流程 (Standard) | Estrella-071 極速版 (Optimized) |
| :--- | :--- | :--- |
| **1. 書本開關次數** | 開關 (傳送前) -> 開關 (讀數據) -> 開關 (去副本)。 | **全程僅開關一次書本**。 |
| **2. 傳送次數** | 2 次 (深塔 + 副本)。 | **1 次** (直達副本，若已完成則僅深塔)。 |
| **3. 異步分析** | 無。 | **在加載黑屏期間**並行處理 OCR。 |
| **4. 結果產出** | 地圖載入完全結束後才顯示。 | 地圖載入到一半時看板已更新。 |

### 結論

上游正式版的邏輯是**順序執行**且**多次傳送回零點**，存在明顯的 UI 交互冗餘與加載等待浪費。
Estrella-071 極速版通過「先快照、後傳送、中分析」的策略，實現了全流程的**時間重疊**，將每日儀式縮減到極限。

### 第三階段：後續副本自動化

| 步驟 | 上游正式版流程 | 優化版流程 (當前) |
| :--- | :--- | :--- |
| **1. 體力校驗** | 傳送完畢 -> 再次開書 -> OCR 讀體力 -> 關書。 | 捕獲傳送前的體力快照，在加載期間已完成校驗。進入副本直接打怪。 |
| **2. UI 穩定性** | 傳送後頻繁出現「找不到書本按鈕」的短暫錯誤。 | 優化了傳送後的書本重測邏輯，增加毫秒級穩定延遲，消除報錯震盪。 |

---

## 優化成果總結

1.  **時間壓縮**：
    - 書本數據獲取從 **~6秒** 壓縮至 **~1.2秒**。
    - 每日首個跳轉（深塔）的「等待體感」減少了約 **3-5秒** 的額外 OCR 阻塞時間。
2.  **流暢度**：
    - 徹底消除了「OCR 沒讀出來所以腳本停在那裡」的尷尬停頓。
    - 利用遊戲本身的加載時間來消化計算量最大的 OCR 指令。
3.  **魯棒性**：
    - 解決了傳送後立即執行操作導致的 UI 競爭問題。
